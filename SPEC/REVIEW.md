# SPEC ↔ PLAN 교차 검증 리포트

## 1. SPEC 요구사항 → PLAN 작업 매핑 누락

### [Critical] 프로젝트 멤버 관리 API/작업 누락
- SPEC 3.1: "멤버/권한 관리" 명시
- PLAN: 프로젝트 API에 멤버 관련 엔드포인트 없음 (GET/POST/DELETE members)
- PLAN Phase 작업에도 멤버 관리 구현 태스크 없음
- **조치 필요**: 프로젝트 멤버 CRUD API 추가 또는 범위에서 제외 결정

### [Warning] 요구사항 기술서 템플릿 다운로드 API 불일치
- SPEC 3.2: 기술서 템플릿도 시스템이 제공
- PLAN API 4.3: `GET /requirements/template/spec` 있음 ✅
- PLAN Phase 작업: Step 1-3에 정제된 엑셀 템플릿만 있고, **기술서 템플릿 다운로드**는 별도 작업으로 안 잡혀있음
- **조치 필요**: Step 1-3 또는 Step 4-2에 기술서 템플릿 생성 작업 포함

### [Warning] 화면설계서 내보내기 API 누락
- SPEC 6.2: 화면설계서 PDF 내보내기 정의됨
- PLAN API 4.4: `GET /features/export` 있음 ✅
- PLAN Phase 5-2: 화면설계서 내보내기 작업 있음 ✅
- 다만 API에서 화면설계서 **개별 파일 다운로드** vs **전체 PDF 묶음** 구분이 안 됨
- **조치 필요**: 화면설계서 내보내기 API 명확화 (개별 vs 묶음)

## 3. API 완전성

### [Critical] 추적성 링크 CRUD API 누락
- SPEC 3.7: 연결/해제 가능 명시
- PLAN DB: TraceabilityLink 테이블 있음 ✅
- PLAN API: 추적성 섹션(4.8)에 **매트릭스 조회만** 있고, **링크 생성/삭제 API가 없음**
- **조치 필요**: 추적성 링크 CRUD API 추가
```
POST   /projects/:id/traceability/links     - 링크 생성
DELETE /projects/:id/traceability/links/:id  - 링크 삭제
```

### [Warning] 테스트 시나리오/케이스 상세 조회 API 누락
- PLAN API 4.6: 시나리오 GET 상세(/:sId), 케이스 GET 상세(/:cId) 엔드포인트 없음
- 목록만 있고 개별 상세 조회가 빠짐
- **조치 필요**: 상세 조회 엔드포인트 추가
```
GET    /projects/:id/test-scenarios/:sId     - 시나리오 상세
GET    /projects/:id/test-cases/:cId         - 케이스 상세
```

### [Warning] 테스트 케이스 삭제 API 누락
- PLAN API 4.6: 케이스 생성/수정/수행은 있으나 **삭제 API**가 없음
- **조치 필요**: DELETE 엔드포인트 추가
```
DELETE /projects/:id/test-cases/:cId         - 케이스 삭제
```

### [Info] 파일 업로드 일반 API 미정의
- 화면설계서 업로드는 기능 API에 있으나, **일반 파일 저장 경로/크기 제한** 미정의
- 구현 시 결정 가능하나 미리 정하면 좋음

## 4. DB 스키마 완전성

### [Critical] Requirement 스키마에 `비고(note)` 필드 누락
- SPEC 5.1 엑셀 템플릿: "비고" 컬럼 정의됨
- SPEC 6.1 산출물 템플릿: "비고" 컬럼 정의됨
- PLAN DB Requirement: `비고` 필드 없음
- **조치 필요**: Requirement 엔티티에 `note (nullable)` 필드 추가

### [Warning] Feature 스키마에 `projectId` 누락
- Task, TestScenario에는 `projectId` 있음
- Feature에는 `reqId`만 있고 `projectId` 직접 참조 없음 (요구사항 경유로 조회 가능하나 쿼리 복잡)
- PLAN DB에 `projectId` 있음 ✅ (확인 결과 있음 — 정상)

### [Warning] User 스키마에 `updatedAt` 누락
- 다른 엔티티에는 대부분 `createdAt/updatedAt` 있으나 User에는 `updatedAt` 없음
- **조치 필요**: 일관성을 위해 추가 권장

### [Info] LLMConfig 스키마에 `updatedAt` 누락
- `createdAt`만 있음. 설정 변경 추적을 위해 `updatedAt` 추가 권장

### [Info] TraceabilityLink에 `createdBy` 누락
- 누가 링크를 만들었는지 추적 불가. 선택적 추가 권장

## 5. 와이어프레임 커버리지

### [Warning] 기능 리스트 목록/상세 와이어프레임 없음
- 요구사항, Task 상세는 와이어프레임 있음
- **기능 리스트 목록/상세** 와이어프레임 없음
- 구현 시 요구사항 상세와 유사한 패턴으로 유추 가능하나, 화면설계서 첨부 섹션이 고유함
- **조치 필요**: 구현 시 참조 가능한 수준이면 생략 가능, 아니면 추가

### [Warning] 테스트 시나리오/케이스 상세 와이어프레임 없음
- 동일. 구현 시 패턴 유추 가능

### [Warning] 관리자 페이지 와이어프레임 없음
- LLM 설정, 사용자 관리, 템플릿 관리 화면 없음
- 표준 CRUD 패턴이라 유추 가능

### [Info] 로그인/회원가입 와이어프레임 없음
- 표준 패턴. 생략 합리적.

## 6. 문서 간 모순/불일치

### [Warning] SPEC 상세화면 예시 vs WIREFRAME 불일치
- SPEC 3.7 예시에서는 **탭 방식** (`[기본 정보] [연결 항목] [버전]`)으로 표시
- WIREFRAME에서는 **섹션 방식** (단일 스크롤)으로 확정됨
- SPEC의 예시가 아직 탭 표기로 남아있음
- **조치 필요**: SPEC 3.7 예시를 섹션 방식으로 수정

### [Warning] SPEC 추적성 체인 표기 불일치
- SPEC 2장: `요구사항 <-> 기능 <-> Task <-> 테스트 시나리오 <-> 테스트 케이스`
- SPEC 3.7 (C안): Task ↔ 테스트 시나리오 직접 연결 **불허**
- 2장의 체인 표기가 모든 인접 노드 간 양방향 연결을 암시하나, 실제 C안에서는 Task↔테스트 불허
- **조치 필요**: 2장 추적성 체인 표기를 C안 규칙에 맞게 수정

### [Info] source 필드 값 불일치
- PLAN DB: `source (excel/manual/md_ai)`
- SPEC: 입력경로 4가지 (엑셀Import/수동생성/마크다운AI/**기술서Import**)
- `spec_ai`가 source enum에 빠져있음
- **조치 필요**: source enum에 `spec_ai` 추가

## 7. 모호한 요구사항

### [Warning] 버전 관리 "전체 스냅샷"의 범위 불명확
- SPEC 3.6: "현재 상태 전체 스냅샷 저장"
- 요구사항 버전 저장 시: 해당 프로젝트의 **모든 요구사항 전체**를 스냅샷? 아니면 **개별 요구사항 1건**?
- 목록 화면에 "버전 저장" 버튼 → 전체 목록 스냅샷으로 보임
- **조치 필요**: 버전 스냅샷 단위 명확화 (산출물 단위 = 전체 목록 스냅샷)

### [Warning] 화면설계서 첨부 형식 미정의
- 허용 파일 형식 (png/jpg/pdf/figma?), 최대 크기, 다중 첨부 여부 미정의
- **조치 필요**: 구현 전 정의 필요

### [Info] 프롬프트 템플릿 "커스터마이징" 범위
- 관리자가 프롬프트를 어디까지 수정 가능한지 불명확
- 전체 프롬프트 자유 편집? 변수 삽입 위치만 편집? 시스템 프롬프트 + 사용자 프롬프트 분리?
- 구현 시 결정 가능

## 8. 구현 차단 요소

### [Critical] 상태 전이(Transition) 규칙 미정의
- 요구사항 상태: 신규 → 검토중 → 확정 → 변경 → 삭제
- Task 상태: 대기 → 진행중 → 완료 → 보류
- **어떤 상태에서 어떤 상태로 전이 가능한지** 규칙 없음
- 예: "확정" → "신규"로 되돌릴 수 있나? "완료" → "진행중" 가능?
- **조치 필요**: 상태 전이 매트릭스 정의 (또는 자유 전이 허용 명시)

### [Warning] 멀티 프로젝트 간 데이터 격리 범위
- 사용자가 여러 프로젝트에 참여 시, 프로젝트 간 데이터 격리 수준 미정의
- 모든 프로젝트에 모든 사용자가 접근 가능? 프로젝트별 멤버 제한?
- **조치 필요**: SPEC 3.1 "멤버/권한 관리"의 상세 정의

### [Warning] 요구사항 ID 자동 생성 규칙 상세
- REQ-001 형태로 자동 생성한다고만 되어있음
- 프로젝트별 독립 채번? 전체 시스템 글로벌? 삭제 후 재사용?
- **조치 필요**: 구현 전 채번 규칙 확정 (권장: 프로젝트별 독립, 삭제 후 미재사용)
